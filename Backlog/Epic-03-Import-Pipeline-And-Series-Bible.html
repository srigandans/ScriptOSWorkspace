#121 Feature 3.1 — Import Pipeline Sprint 4
The Import Service accepts FDX, Fountain, PDF, and scanned PDF files and converts them to the canonical Script AST via a five-stage pipeline: format detection → parser → normalization → confidence scoring → provenance record. #129 User Story As a developer the Import Saga orchestrates the full import lifecycle end-to-end via Temporal with human-in-the-loop review and batch zip support
Sprint 4 ● Critical 13 pts temporal import-saga workflow
The Import Saga is a Temporal workflow that sequences all import pipeline steps reliably, handles failures with compensation, and pauses at the human review step for low-confidence imports. Temporal's durable execution guarantees that an import is never lost even if the worker pod restarts mid-import. The batch saga enables showrunners to import an entire season's worth of scripts from a zip file in a single operation with episode number auto-inference.
Acceptance Criteria

  • End-to-end import of a single FDX file completes in under 60s (excluding review wait time)
  • Human review step: saga pauses at WaitForSignal activity; writer sends reviewComplete signal with accepted/corrected element list; saga resumes; rejected elements excluded from persist
  • If worker pod restarts during any step: Temporal replays from last completed activity; no data lost; no duplicate imports
  • Batch import saga: accepts zip file S3 key; extracts files; runs a child workflow per file; episode numbers inferred from filename patterns (e.g. 1x01, S01E01, episode_1); completes 5 episodes in under 3 minutes
  • Import Saga Temporal worker deployed as workers/import-saga/ Kubernetes Deployment

#140 Feature 3.3 — Series Bible Service Sprint 5
The Series Bible Service manages a Neo4j Aura graph of characters, locations, props, world rules, story arcs, and lore facts. Every entity has a stable UUID, is project-scoped, and participates in a rich relationship graph (APPEARS_IN, SET_IN, HAS_VOICE, SUPERSEDES, RELATED_TO, PARTICIPATES_IN). The canon consistency checker uses graph traversal to detect contradictions between new facts and established ones. Voice profiles per character capture vocabulary, speech patterns, and forbidden words for AI grounding. Stub entities are auto-created when character and location names are detected on script checkpoint.
Reference: Wiki/03-data-model.md Series Bible Graph, Wiki/15-roadmap.md Sprint 5.

  • APPEARS_IN and SET_IN relationships created when character/location first linked to a script scene
  • Fact supersedure chain traversable: SUPERSEDES relationship queryable via Cypher path query
  • Canon consistency check: validateScriptConsistency gRPC returns ConflictResult[] for a contradictory test case
  • Voice profile stored with vocabularyLevel, speechPatterns, forbiddenWords, referenceSceneIds
  • AI context gRPC methods (getWorldRules, getCharacterFacts, getVoiceProfiles) all returning correct data
  • Character and location stub nodes auto-created from script.checkpointed NATS events within 30s

  • All nodes carry: id (UUID), orgId, projectId, createdAt, updatedAt, isActive
