#70 Feature 2.1 — Script Service Core Sprint 2
The Script Service is the authoritative data store for all screenplay content. It owns the script.* PostgreSQL schema (projects, scripts, ast_nodes, script_versions) managed via Atlas. It exposes a GraphQL subgraph and a gRPC API used by the Collaboration Service for checkpoints and by the Temporal publish-saga for locking. It publishes the script.checkpointed NATS event that triggers Breakdown and Search re-processing. This is the first Tier 1 service deployed after Auth; every downstream service depends on it.
Reference: Wiki/15-roadmap.md Sprint 2, Wiki/03-data-model.md §PostgreSQL DDL, Wiki/16-system-design.md §Script Service.
Feature Acceptance Criteria

  • upsertASTNodes gRPC batch upsert succeeds for up to 500 nodes in under 500ms
  • getSceneSnapshot returns heading, element counts, and page estimate
  • createVersion stores CRDT snapshot; updates scripts.current_version_id
  • lockDraft / unlockDraft guard the publish-saga lifecycle with row-level locking
  • script.checkpointed NATS event published with scriptId, orgId, changedSceneIds, versionId
  • Script Service Docker image deployed to staging; /health returns 200
  • Atlas migration for script.* schema applied cleanly to staging Postgres

#104 User Story As writers, structural screenplay operations (move, split, and merge scenes) remain consistent across all connected clients
Sprint 3 ● Critical 8 pts structural-ops redis-lock adr-019
Writers frequently reorganise scripts by moving, splitting, or merging scenes. These structural operations are handled as atomic Loro MovableTree operations with a Redis coordination lock to prevent concurrent conflicts that CRDT alone cannot resolve deterministically. Implements ADR-019.
Acceptance Criteria
  • Scene move while another writer types: after sync all clients see scene at new position with typing intact
  • Concurrent different-scene moves: both moves applied; no scenes lost
  • Concurrent same-scene moves: Redis lock ensures one wins; losing client returns CONCURRENT_STRUCTURAL_OP and sees winner's move within 500ms
  • Scene split after child 3 of 5: original scene retains children 1-3; new scene has children 4-5 + empty heading; all clients converge within 500ms
  • Scene merge: concatenated content in correct order; source scene removed; all clients converge within 500ms
  • All structural ops represented as discrete AST mutations in the next checkpoint cycle
