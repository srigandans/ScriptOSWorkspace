#1 Epic Platform Foundation
Sprint 0–1 ● Priority: Critical platform foundation infrastructure sprint-0 sprint-1
Description
Establish the complete development infrastructure, cloud hosting environment, authentication system, shared TypeScript packages, and API gateway that ALL subsequent service epics depend on. This epic covers Sprints 0 and 1 (Weeks 1–4). No end-user features are delivered; the output is a fully operational development and staging environment with a working authentication and authorisation flow. Completion of this epic is the hard gate for starting Epic 2 onwards.

Reference: Wiki/15-roadmap.md Sprint 0 and Sprint 1.
Acceptance Criteria (Epic-level Definition of Done)
  • Every engineer can clone the repo and run docker compose up in under 2 minutes
  • GKE Autopilot staging cluster operational with Istio mTLS enforced
  • All Tier 0 data stores running in staging (Postgres, Redis, NATS, Neo4j, Elasticsearch, Temporal, S3)
  • Auth Service end-to-end: Google OAuth login returns valid RS256 JWT
  • Apollo Router federated gateway running; JWT validation enforced at gateway
  • All shared packages (ast, events, proto, common, db) compile with zero TypeScript errors
  • CI pipeline green on main branch (lint, typecheck, test, Docker build)

#40 User Story As a platform engineer, all API requests carry RBAC role claims so downstream services can enforce authorisation without querying the database
Sprint 1 ● Critical 5 pts
Downstream services must never make a database call to check if a user is allowed to perform an action — this creates N+1 auth DB queries at scale. The JWT carries the user's roles and org ID. The HTTP Gateway validates the JWT and injects X-User-ID, X-Org-ID, X-Roles, X-Tier headers. Row-level security in Postgres uses app.org_id session variable for tenant isolation. Reference: Wiki/16-system-design.md Auth Boundary, ADR-012.
Acceptance Criteria
  • JWT payload includes: sub, org, roles (array), tier, iat, exp
  • HTTP Gateway rejects requests without valid JWT with 401
  • HTTP Gateway injects X-User-ID, X-Org-ID, X-Roles, X-Tier headers into all forwarded requests
  • Integration test: writer role cannot call producer-only mutation; receives 403 from the service
  • Postgres RLS: SELECT from tenant table without setting app.org_id returns zero rows
  • gRPC internal calls include user context as metadata (x-user-id, x-org-id, x-roles)
